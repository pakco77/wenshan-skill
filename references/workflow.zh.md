# 文山.skill 中文工作流

![文山.skill 中文与 English 效果对照](../assets/demo-bilingual.png)

**用山脉展示你的篇章。**

## 输入

- 作者昵称：用于页眉与传播图署名。
- 文章集合：一个 Obsidian 文件夹、Markdown 文件夹或明确文章清单。
- 界面语言：`zh`。

只分析用户选定的集合，不默认扫描整个 Vault。

## 三段式流水线

### 1. 语义判断

让本地 Agent 逐篇形成可审计卡片。判断文章是否包含用户自己的持久判断，排除提示词、操作说明、模板、第三方样例、空文件和脏数据。保留原文标题与源路径。

### 2. 版本归并与规则抽离

识别草稿、成稿和改写版本，设置 `version_family`，每族只选一张 canonical。

比较所有 canonical 判断，寻找被多篇文章反复表达的因果关系、选择原则或决策规律。山名必须是 Agent 从当前材料中抽出的规则，不是用户预设主题，也不是“AI工具”“产品经理”之类分类词。

好的规则：

- 能离开原文章仍然成立。
- 表达一个关系，而不是一个对象。
- 可以被证据支持或反驳。
- 对别的用户和别的知识库仍有解释力。
- 中文通常控制在 4–12 个字。

例如：

| 证据域 | 不合格山名 | 合格规则山名 |
|---|---|---|
| AI工具 | AI工具 | 摩擦决定采用 |
| AI认知 | AI认知 | 能力普及，判断升值 |
| 产品经理 | 产品经理 | 方向判断先于执行 |

每条规则至少由三篇独立 canonical 文章支撑。没有证据就没有山，不显示“待勘探”。

### 3. 地图生成

主标题显示抽离规则；小字显示可选证据域；副标题解释这批文章为什么共同支持该规则。篇数决定海拔和山体量。

双语字段使用 `rule`、`explanation` 与经审阅的 `rule_en`、`explanation_en`。证据文章标题保持原文。

```bash
python3 scripts/render_territory_demo.py \
  --scope "/绝对路径/文章集合" \
  --nickname "作者昵称" \
  --language zh \
  --output-name "文山"
```

## 地图读法

- 主标题：从当前文章集合抽离出的规则。
- 证据域：帮助理解材料来自哪里，不决定山名。
- 副标题：Agent 对该规则为何成立的证据解释。
- `13篇`：由 13 篇独立 canonical 文章形成的海拔。
- 点：文章证据，点击后返回原文。
- 等高线：规则内部的证据聚集形状。
- 跨山关系：可以作为桥显示，但不重复增加第二座山高度。

## 发布前校验

- 山名不是主题、工具、供应商或文件夹名。
- 每座山至少三条唯一 canonical 源路径。
- 各山篇数之和等于地图独立文章数。
- 不显示本地绝对路径、调试标签或虚构候选山。
- 13 寸屏可完整查看；1080×1440 截图不裁山、不带控制按钮。
